# 🛡️ 開発・運用セーフティ＆クオリティ・ガーディアン

このスキルは、システムへの変更、コードの修正、またはパッケージのインストールを行う際に、安全性を確保し、品質を維持するための標準手順を定義します。

## 🎯 適用条件 (Trigger)
- ユーザーから「コードの修正」「機能追加」「バグ修正」を依頼されたとき
- サーバー設定の変更やパッケージのインストールを依頼されたとき
- ファイルの削除や移動など、破壊的な操作が含まれるとき

## 📋 実行手順 (Steps)

### 1. 現状把握 (Context Discovery)
変更を加える前に、必ず以下の情報を確認してください。
- 現在のディレクトリ構成と主要なファイルの確認。
- 修正対象となるファイルの既存コードの読み込み。
- (サーバーの場合) OSの種類、実行権限、依存関係の確認。

### 2. 変更計画の提示 (Planning)
実際に手を動かす前に、ユーザーに対して「何をするか」を要約して伝えてください。
- 修正する箇所の特定。
- 導入するライブラリやコマンドの影響範囲。
- **[重要]** もし破壊的な変更（削除や上書き）がある場合は、必ずバックアップ（`.bak`作成やGitコミット）を提案してください。

### 3. 最小単位での実行 (Atomic Execution)
- 一度に大量のファイルを変更せず、論理的な単位ごとにステップを分けて実行してください。
- コードを生成する際は、プロジェクトの既存の命名規則やスタイル（インデント等）に厳格に従ってください。

### 4. 検証とテスト (Verification)
変更直後に以下のいずれかを実施してください。
- 構文チェック（linter）の実行。
- 関連するテストコードがある場合はその実行。
- サーバー設定なら、設定ファイルのバリデーションコマンド（例: `nginx -t`）の実行。

### 5. 完了報告 (Reporting)
- 何を変更したか。
- 動作確認の結果はどうだったか。
- 次にユーザーが確認すべきことは何か。

## ⚠️ 禁止事項 (Constraints)
- ユーザーの明示的な許可なく `rm -rf /` や `rm *` などの広範囲な削除コマンドを実行しない。
- 既存のコメントや重要なドキュメントを勝手に削除しない。
- 解決策が複数ある場合は、独断せずユーザーに選択肢を提示する。
